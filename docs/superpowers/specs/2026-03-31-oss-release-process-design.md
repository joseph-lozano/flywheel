# OSS Release Process Design

## Context

Flywheel is moving from a private source repo + public `joseph-lozano/flywheel-releases` distribution repo to a single open-source repository. All release artifacts, changelogs, and update manifests will live in the main repo. No migration is needed — all existing releases are pre-release alpha and the only current user is the maintainer.

## Goals

- Consolidate releases into the main (now public) repo
- Replace manual changelog management with Changesets
- Remove the now-unnecessary release notes cleaning script (it existed to strip author/PR links from the private repo)
- Simplify the release workflow

## Changesets Setup

Add `@changesets/cli` as a dev dependency. Initialize with `.changeset/config.json`:

```json
{
  "$schema": "https://unpkg.com/@changesets/config/schema.json",
  "changelog": "@changesets/changelog-github",
  "commit": false,
  "linked": [],
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

Note: `@changesets/changelog-github` requires a `GITHUB_TOKEN` env var to be set when running `pnpm changeset version` locally — it uses it to look up PR details and enrich changelog entries with links. Without it, changeset falls back to a plain format. Add to local shell profile or pass inline: `GITHUB_TOKEN=<pat> pnpm changeset version`.

Add scripts to `package.json`:

```json
{
  "changeset": "changeset",
  "changeset:version": "changeset version"
}
```

**Contributor workflow:** Run `pnpm changeset` during a PR to generate a `.changeset/*.md` file describing the change (patch/minor/major + description). Commit the file alongside the PR changes.

**Release workflow (manual):**
1. Run `pnpm changeset version` — bumps `package.json`, writes/updates `CHANGELOG.md`, deletes consumed changeset files
2. Review `CHANGELOG.md`
3. Commit (`chore: release vX.Y.Z`), tag (`git tag vX.Y.Z`), push both

## electron-builder.yml

Change the publish provider repo from `joseph-lozano/flywheel-releases` to `joseph-lozano/flywheel`. No other publish config changes needed.

## Release Workflow (.github/workflows/release.yml)

### Remove

- All steps that clone `joseph-lozano/flywheel-releases`, push CHANGELOG there, or copy release notes to it
- Usage of `scripts/clean-release-notes.js`
- The `GH_TOKEN` secret (PAT scoped to the releases repo) — replace with `GITHUB_TOKEN` for same-repo publishing

### Keep / Update

- The `release-notes` job continues to create a GitHub Release in the main repo
- Extract the relevant version section from `CHANGELOG.md` (generated by Changesets) as the release body — this keeps release notes and CHANGELOG consistent

## scripts/clean-release-notes.js

Delete this file. It was only needed because the source repo was private and links/authors would be meaningless to external users. With the repo public, GitHub's auto-generated notes are appropriate as-is.

## README.md

Replace the "Releasing" section with the Changesets workflow:

1. During PR work: `pnpm changeset` — describe your change, commit the `.changeset` file
2. When ready to release: `pnpm changeset version` — bumps version, generates CHANGELOG
3. Commit, tag, push: `git tag vX.Y.Z && git push origin main vX.Y.Z`

Remove references to `flywheel-releases` repo and the `GH_TOKEN` secret.

## CHANGELOG.md

Will be initialized in the main repo on the first `pnpm changeset version` run. No manual initialization needed.

## Secrets / CI Changes

- Remove `GH_TOKEN` from repository secrets (was a PAT for writing to flywheel-releases)
- `GITHUB_TOKEN` (auto-provided by Actions) is sufficient for publishing releases to the same repo
- All other secrets (code signing, notarization) remain unchanged
